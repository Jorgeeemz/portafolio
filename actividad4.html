🛡️ Actividad 4 – Parcial 1

Mecanismos de defensa de red: configuración de firewall con iptables

Jorge Alejandro Cabrera Meza – 181591 Febrero 2026 Políticas · Filtrado · Stateful firewall

Descarga un respaldo académico de esta actividad.

1. Introducción contextual

Los mecanismos de defensa de red constituyen la primera línea de protección contra accesos no autorizados y amenazas externas. Dentro de estos, el firewall es el componente esencial que inspecciona y filtra el tráfico basándose en reglas predefinidas. En entornos Linux, iptables sigue siendo la herramienta estándar para implementar firewalls estatales (stateful), permitiendo definir políticas de filtrado por interfaz, dirección IP, puerto, protocolo y estado de conexión.

Esta actividad práctica consiste en diseñar y escribir reglas iptables para escenarios típicos: establecer políticas restrictivas, permitir tráfico legítimo (web, correo, DNS) y gestionar conexiones establecidas. El objetivo es desarrollar la capacidad de traducir requisitos de seguridad a comandos precisos y funcionales.

A continuación se analizan las reglas propuestas, se identifican errores sintácticos y de lógica, y se presentan las versiones corregidas junto con una explicación detallada.

2. Desarrollo técnico: análisis y corrección de reglas

La siguiente tabla muestra las reglas propuestas en la actividad original. Se ha respetado la redacción del estudiante y se ha añadido una columna con la versión corregida y una explicación técnica.

Regla (descripción) Comando sugerido (original) Comando corregido Observaciones / Corrección
Establecer una política restrictiva. Iptables -p FORWARD DROP iptables -P FORWARD DROP La opción -P (mayúscula) establece la política por defecto. -p es para protocolo.
Permitir el tráfico de conexiones ya establecidas. Iptables -A -m FORWARD state -state ESTABLISHED, RELATED -j ACCEPT iptables -A FORWARD -m state --state ESTABLISHED,RELATED -j ACCEPT El módulo es state, no va separado; los guiones dobles en --state. La cadena debe ir después de -A.
Aceptar tráfico DNS (TCP) saliente de la red local. Iptables -A INPUT -p tcp -s 192.1.2.11 -d 0.0.0.0 --dport 53 -m state --state NEW -j ACCEPT iptables -A FORWARD -p tcp -s 192.1.2.11 --dport 53 -m state --state NEW -j ACCEPT El tráfico que atraviesa el firewall debe ir en FORWARD, no en INPUT. Además, -d 0.0.0.0 es redundante (equivale a cualquier destino).
Aceptar correo entrante proveniente de Internet en el servidor de correo. Iptable -A INPUT -p tcp -s 0.0.0.0 -d 192.1.2.10 --dport 25 -m state -state NEW -j ACCEPT iptables -A FORWARD -p tcp -d 192.1.2.10 --dport 25 -m state --state NEW -j ACCEPT Si el servidor está detrás del firewall, la regla va en FORWARD. Corrección de sintaxis (--state, -j).
Permitir correo saliente a Internet desde el servidor de correo. Iptables -A FORWARD -p tcp -s 192.1.2.10 -d 0.0.0.0 --dport 25 -m state --state NEW -j ACCEPT iptables -A FORWARD -p tcp -s 192.1.2.10 --dport 25 -m state --state NEW -j ACCEPT Correcto en cadena, pero sobra -d 0.0.0.0. Se acepta la regla con ajuste menor.
Aceptar conexiones HTTP desde Internet a nuestro servidor web. Iptables -A FORWARD -p tcp -s 0.0.0.0 -d 192.1.2.11 --dport 80 -m state --state NEW -j ACCEPT iptables -A FORWARD -p tcp -d 192.1.2.11 --dport 80 -m state --state NEW -j ACCEPT Similar al anterior: origen implícito cualquiera, se elimina -s 0.0.0.0.
Permitir tráfico HTTP desde la red local a Internet. Iptables -A FORWARD -p tcp -s 192.1.2.11 -d 0.0.0.0 --dport 80 -m state --state NEW -j ACCEPT iptables -A FORWARD -p tcp -s 192.1.2.11 --dport 80 -m state --state NEW -j ACCEPT Idem, se elimina -d 0.0.0.0. La regla es funcional.

🔍 Análisis de errores comunes detectados

  • Uso incorrecto de mayúsculas: Iptables (con I mayúscula) no es el comando real; debe ser iptables.
  • Políticas vs. reglas: Confundir -P (política) con -p (protocolo).
  • Cadenas equivocadas: Tráfico que cruza el firewall (red local → Internet) debe estar en la cadena FORWARD, no en INPUT (que es para tráfico destinado al propio firewall).
  • Sintaxis del módulo state: Faltan guiones dobles (--state) y a veces se escribe -state.
  • Dirección 0.0.0.0: En iptables, omitir -s o -d equivale a cualquier dirección; escribir -s 0.0.0.0 no es incorrecto pero es redundante y puede confundir.

Todas las correcciones han sido validadas en un entorno de prueba con iptables v1.8.7.

4. Reflexión técnica

Esta actividad evidencia una comprensión general de los conceptos de firewall, pero también revela debilidades en la sintaxis exacta de iptables. Errores como usar -p en lugar de -P para políticas, o colocar reglas de tráfico reenviado en la cadena INPUT, son muy comunes en etapas de aprendizaje. Lo importante es que se identifica la necesidad de permitir tráfico establecido y relacionado y de restringir políticas por defecto, lo cual es la base de un firewall stateful.

Desde mi perspectiva, dominar iptables requiere práctica en consola real y no solo ejercicios escritos. Al ejecutar los comandos, el feedback inmediato (errores de sintaxis, contadores de paquetes) refuerza el aprendizaje. Además, trabajar con direcciones IP concretas (192.1.2.10/11) ayuda a visualizar la topología, pero se debe evitar el uso de 0.0.0.0 como comodín; en iptables, omitir -s o -d ya implica "cualquiera".

En un entorno profesional, una regla mal escrita puede abrir brechas de seguridad o denegar servicios legítimos. Por eso, actividades de corrección como esta son fundamentales: aprender del error es más valioso que acertar a la primera.

— Jorge Alejandro Cabrera Meza (181591).
