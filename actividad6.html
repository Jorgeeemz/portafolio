🔐 Actividad 6 – Parcial 1

Implementación de una VPN Site‑to‑Site con IPSec

Jorge Alejandro Cabrera Meza – 181591 Febrero 2026 IPSec · IKE · ESP · Túnel VPN

Descarga un respaldo académico de esta guía de configuración.

1. Introducción contextual

IPSec (Internet Protocol Security) es un conjunto de protocolos que proporciona seguridad a nivel de red (capa 3 del modelo OSI) mediante cifrado, autenticación e integridad de los datos. Una VPN (Virtual Private Network) basada en IPSec permite crear un túnel seguro a través de una red pública (como Internet), conectando dos redes locales (LAN) de forma transparente, como si estuvieran en el mismo segmento físico.

En entornos corporativos, las VPN site‑to‑site son esenciales para interconectar sedes de manera segura, garantizando la confidencialidad, integridad y autenticidad de la información transmitida. Esta actividad aborda la configuración paso a paso de un túnel IPSec entre dos routers Cisco, explicando desde los conceptos teóricos (AH, ESP, modos transporte/túnel, IKE) hasta los comandos específicos y su verificación.

El objetivo es que el estudiante comprenda no solo la sintaxis de los comandos, sino también el diálogo estructurado en dos fases que establecen los routers para negociar los parámetros de seguridad y proteger los datos sensibles.

2. Desarrollo técnico

2.1. Protocolos de seguridad: AH y ESP

  • AH (Authentication Header): Proporciona autenticación e integridad de los datos, pero no cifrado. Garantiza el origen legítimo y que los datos no fueron modificados.
  • ESP (Encapsulating Security Payload): Protocolo más común. Ofrece cifrado (confidencialidad), autenticación e integridad. En la configuración se usa esp-aes 256 esp-sha-hmac.

2.2. Modos de funcionamiento: transporte y túnel

  • Modo Transporte: Solo la carga útil del paquete IP es cifrada/autenticada; la cabecera original permanece. Uso típico en comunicaciones extremo a extremo.
  • Modo Túnel: Todo el paquete IP (cabecera + datos) es cifrado y encapsulado en un nuevo paquete. Es el modo estándar para VPN site‑to‑site.

2.3. El proceso IKE (Internet Key Exchange)

IKE gestiona el establecimiento del túnel en dos fases:

  • Fase 1 (ISAKMP): Crea un canal seguro de control. Los routers se autentican (clave precompartida secretkey) y acuerdan parámetros (cifrado AES 256, autenticación pre‑share, grupo Diffie‑Hellman 5). El resultado es una ISAKMP SA.
  • Fase 2 (IPSec): Negocia el túnel de datos. Se define el tráfico a proteger (ACL), el transform‑set (esp-aes 256 esp-sha-hmac), y se crean las IPSec SAs (una para cada dirección).

2.4. Topología de red

LAN A 192.168.1.0/24
R1 209.165.100.1

Dos redes locales (192.168.1.0/24 y 192.168.3.0/24) conectadas a través de un ISP mediante un túnel IPSec.

2.6. Configuración IPSec en R1 y R2

R1

access-list 100 permit ip 192.168.1.0 0.0.0.255 192.168.3.0 0.0.0.255
crypto isakmp policy 10
 encryption aes 256
 authentication pre-share
 group 5
crypto isakmp key secretkey address 209.165.200.1
crypto ipsec transform-set R1->R2 esp-aes 256 esp-sha-hmac
crypto map IPSEC-MAP 10 ipsec-isakmp
 set peer 209.165.200.1
 set pfs group5
 set transform-set R1->R2
 match address 100
int g0/0
 crypto map IPSEC-MAP

R2

access-list 100 permit ip 192.168.3.0 0.0.0.255 192.168.1.0 0.0.0.255
crypto isakmp policy 10
 encryption aes 256
 authentication pre-share
 group 5
crypto isakmp key secretkey address 209.165.100.1
crypto ipsec transform-set R2->R1 esp-aes 256 esp-sha-hmac
crypto map IPSEC-MAP 10 ipsec-isakmp
 set peer 209.165.100.1
 set pfs group5
 set transform-set R2->R1
 match address 100
int g0/0
 crypto map IPSEC-MAP

2.7. Explicación detallada de comandos clave

ComandoContextoExplicación
access-list 100 permit ip 192.168.1.0 0.0.0.255 192.168.3.0 0.0.0.255 ACL Define el "tráfico interesante". Todo el tráfico IP de la red 192.168.1.0/24 con destino a 192.168.3.0/24 será protegido por IPSec.
crypto isakmp policy 10 IKE Fase 1 Crea una política ISAKMP con prioridad 10. Se negocia el canal de control.
encryption aes 256 IKE Fase 1 Especifica AES con clave de 256 bits para cifrar la negociación IKE.
authentication pre-share IKE Fase 1 Utiliza una clave precompartida para autenticar los peers.
group 5 IKE Fase 1 Grupo Diffie‑Hellman 5 (DH de 1536 bits) para el intercambio seguro de claves.
crypto isakmp key secretkey address 209.165.200.1 IKE Fase 1 Clave precompartida secretkey para el peer remoto con IP 209.165.200.1.
crypto ipsec transform-set R1->R2 esp-aes 256 esp-sha-hmac IPSec Fase 2 Crea un transform‑set que define el protocolo ESP con cifrado AES‑256 y autenticación HMAC‑SHA.
crypto map IPSEC-MAP 10 ipsec-isakmp Crypto Map Entidad que une todos los elementos: peer, transform‑set, ACL, PFS.
set pfs group5 Crypto Map Activa Perfect Forward Secrecy usando DH grupo 5 en la Fase 2.
int g0/0 / crypto map IPSEC-MAP Interfaz Aplica el crypto map a la interfaz de salida (hacia el ISP), activando el túnel.

2.8. Verificación del túnel

  • show crypto isakmp sa – Muestra el estado de la Fase 1. Un estado QM_IDLE o ACTIVE indica que IKE se estableció correctamente.
  • show crypto ipsec sa – Muestra las SA de la Fase 2. Deben aparecer paquetes encapsulados/desencapsulados y los SPI.
  • show ip route – Verifica que las rutas a las redes locales y remotas existen.

Ejemplo de salida esperada para show crypto ipsec sa:

3. Material multimedia

🧩 Diagrama de flujo IKE (Fase 1 y Fase 2)

Fase 1 (ISAKMP)

  1. Negociación de políticas
  2. Intercambio DH (claves)
  3. Autenticación (pre‑share)
  4. → ISAKMP SA establecida

Fase 2 (IPSec)

  1. Negociación transform‑set
  2. Intercambio DH (PFS opcional)
  3. Creación de IPSec SAs
  4. → Túnel de datos activo

4. Reflexión técnica

La implementación de una VPN site‑to‑site con IPSec demuestra cómo la teoría de redes se aplica directamente en la práctica. No se trata solo de ejecutar comandos, sino de comprender el diálogo estructurado en dos fases (IKE) que establecen los routers para acordar cómo protegerse mutuamente (Fase 1) y cómo proteger los datos sensibles (Fase 2).

Detalles como el grupo Diffie‑Hellman, el algoritmo de cifrado (AES‑256) y la activación de PFS no son meros tecnicismos; son decisiones de diseño que determinan la robustez del túnel frente a ataques. Por ejemplo, PFS garantiza que si una clave de sesión se ve comprometida, las claves anteriores y futuras no se ven afectadas.

En entornos profesionales, dominar IPSec es esencial para garantizar la confidencialidad, integridad y autenticidad de la información que viaja entre sedes. Esta práctica refuerza la capacidad de diagnosticar problemas de conectividad (usando show crypto isakmp sa y show crypto ipsec sa) y de ajustar parámetros según los requisitos de seguridad de la organización.
